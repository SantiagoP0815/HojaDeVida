Análisis Detallado del Modelo de Datos

Este diagrama sigue un modelo relacional normalizado centrado en la entidad PERSONAS, diseñado para garantizar la integridad y evitar la redundancia de datos:

  • Centralidad de PERSONAS: La tabla PERSONAS actúa como el núcleo del sistema. Contiene la información inmutable y de identificación del usuario (documento, nombre, contacto). Todas las demás tablas son satélites que enriquecen este perfil.
  • Relaciones 1 a 1 (||--||):
    • EDUCACION_BASICA_MEDIA: Se define como 1 a 1 porque, en el contexto de este sistema, un usuario registra un único bloque de información sobre su bachillerato o educación media. No se espera un historial múltiple de graduaciones de colegio.
    • TIEMPO_EXPERIENCIA: Es una tabla de resumen agregada. Cada persona tiene un único registro que totaliza sus años y meses de experiencia en diferentes sectores. Esto optimiza las consultas al evitar recalcular totales cada vez que se necesita ver el perfil.
  • Relaciones 1 a Muchos (||--o{):
    • EDUCACION_SUPERIOR, IDIOMAS, EXPERIENCIA_LABORAL: Estas entidades permiten múltiples registros por usuario. Una persona puede tener varios títulos universitarios, hablar varios idiomas o haber trabajado en múltiples empresas.
    • DOCUMENTOS: Permite asociar múltiples archivos de soporte (PDFs, imágenes) a una sola persona, manteniendo los metadatos de los archivos (nombre, ruta, tamaño) separados de la lógica de negocio.
  • Integridad Referencial: Todas las tablas satélite tienen una llave foránea (FK) id_persona, lo que garantiza que no pueda existir información de educación o experiencia sin un usuario asociado.

Explicación del Flujo de Procesamiento

Este diagrama ilustra cómo el sistema maneja la complejidad de recibir datos de texto y archivos binarios simultáneamente, garantizando la consistencia de los datos:

  • Manejo de Multipart/Form-Data: El navegador envía los datos en un formato multipart. El componente Middleware (Multer) es el primero en intervenir, interceptando la solicitud para procesar y guardar los archivos físicos en el servidor antes de que la lógica de negocio se ejecute. Esto separa la responsabilidad de I/O (entrada/salida) de la lógica de aplicación.
  • Validación Previa: Antes de tocar la base de datos, express-validator asegura que los campos obligatorios (como documento o nombres) estén presentes y tengan el formato correcto. Esto protege al sistema de datos corruptos o incompletos.
  • Transaccionalidad (ACID): El bloque azul rect representa una Transacción de Base de Datos.
    • La operación de guardar una hoja de vida implica insertar datos en 5 o 6 tablas diferentes.
    • Si falla la inserción de la experiencia laboral (por ejemplo, por un error de sintaxis), la transacción hace ROLLBACK. Esto significa que se deshacen los cambios en PERSONAS y EDUCACION, evitando que quede un usuario "a medias" en el sistema. Todo se guarda o nada se guarda.
  • Orquestación en Capas:
    • El Controller solo coordina: recibe la petición y decide qué servicio llamar.
    • El Service contiene la lógica "dura": sabe en qué orden guardar los datos y maneja la transacción.
    • El Repository ejecuta el SQL puro, manteniendo el código SQL aislado de la lógica de negocio.

Arquitectura de Software (MVC + Capas)

El sistema implementa una arquitectura modular basada en el patrón Modelo-Vista-Controlador (MVC) extendido con capas de servicio y repositorio para mayor mantenibilidad:

  • Capa de Presentación (Vista): Utiliza Handlebars (.hbs) para renderizar HTML dinámico en el servidor. Esto permite mostrar datos personalizados al usuario de manera eficiente. Los archivos estáticos (CSS, JS) se sirven directamente para la interactividad del cliente.
  • Capa de Control (Controlador): Es el punto de entrada.
    • Rutas: Definen los endpoints (URLs) disponibles.
    • Middleware: Maneja aspectos transversales como la autenticación (Passport.js), la subida de archivos (Multer) y la seguridad, asegurando que solo peticiones válidas lleguen a los controladores.
  • Capa de Negocio (Servicio): Aquí reside la inteligencia del sistema. HojaService no sabe de HTTP (req/res) ni de SQL. Solo sabe de reglas de negocio: "Para actualizar una hoja de vida, primero debo validar los archivos, luego iniciar una transacción, etc.". También integra librerías externas como PDFMake para generar documentos.
  • Capa de Datos (Modelo/Repositorio): Abstrae el acceso a la base de datos. HojaRepository contiene todas las consultas SQL. Si mañana cambiamos de MySQL a PostgreSQL, idealmente solo tendríamos que modificar esta capa, sin tocar los controladores ni las vistas.
